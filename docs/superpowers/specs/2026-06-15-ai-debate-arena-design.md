# AI 辩论台（AI Debate Arena）— 设计文档

> 一个桌面 App：用户输入一个话题、选两个 AI，系统自动分配正/反方，让两个 AI 在各自的**网页版**里来回辩论 N 回合。全程白嫖各 AI 网页版的免费账号，**不调用任何付费 API**。

- 日期：2026-06-15
- 形态：桌面 App（Electron），**fork 自 [ChatALL](https://github.com/ai-shifu/ChatALL)**
- 阶段：第一期（MVP）

---

## 1. 背景与关键结论

需求：输入一个话题，让两个 AI 自动连续辩论；想"白嫖"（不充钱、不持有付费 key、用各家免费网页版）。

技术调研结论（已实测）：
- **做成网站行不通**。主流 AI 站点（ChatGPT、DeepSeek、Google 等）通过 `X-Frame-Options` / `CSP: frame-ancestors` 禁止被 iframe 嵌入；即便能嵌，浏览器**同源策略**也禁止页面跨域读写 iframe 内容，无法"自动输入 + 读取回答 + 把一方的话喂给另一方"。
- **App / 浏览器插件可行**。它们能控制自己的浏览器内核、向目标网页注入脚本，绕过上述限制。**ChatALL 已经验证了这条路**：Electron 桌面 App，通过 "Web Access" 驱动 40+ 个 AI 网页版，很多无需 API key。
- 决策：**桌面 App，fork ChatALL**，在其"网页版驱动内核"之上加一层"辩论编排器"。

## 4. 辩论编排器（核心逻辑）

### 4.1 利用网页版自带的上下文记忆
为每个 AI 各开**一个对话窗口**，全程在**同一个窗口**里追问。因此每个 AI 天然记得自己之前的发言与脉络。编排器每轮**只需把"对方刚说的话"注入进去**（对手在另一个窗口，AI 默认看不到）。

### 4.2 提示词模板（自动套用，按语言环境生成对应语言版本）

**第 1 回合 · 正方 A 开场（确立立场，等价于 system prompt）：**
> 我们进行一场辩论，辩题：「{话题}」。你是正方，立场是支持该观点。请用有力论据陈述开场，约 200 字，像真人辩手一样，直接开始，不要客套。

**第 1 回合 · 反方 B（注入 A 的话）：**
> 我们进行一场辩论，辩题：「{话题}」。你是反方，立场是反对该观点。对方（正方）刚才说：『{A1}』。请针对性反驳并陈述观点，约 200 字。

**第 2 回合起 · 只注入对方上一句：**
> 给 A：对方（反方）回应：『{B_上一轮}』。请继续反驳、强化论点，约 200 字。
> 给 B：对方说：『{A_本轮}』。请继续反驳。

### 4.3 传话循环
```
输入话题 T → 识别语言 L
轮 1：模板1(T,L) → A → A1 ; 模板2(T,A1,L) → B → B1
轮 k（2..N）：模板3(B_{k-1}) → A → A_k ; 模板3(A_k) → B → B_k
N 轮后：界面显示"辩论结束"
```
- 每轮需检测"AI 是否回答完毕"（复用 ChatALL 已有的完成检测）后再传话。

### 4.4 语言识别
- 对用户输入的话题文本做轻量语言检测（中/英等）；命中则用对应语言的模板，否则回退系统 locale。

### 4.5 登录检测（交互关键环节）
白嫖网页版 = 依赖用户自己的网页账号，因此开辩前后必须检测登录态，避免注入卡在登录墙。

**核心逻辑（简化）：**
- **检测方式**：读取该 webview 当前页面的 HTML/DOM 源码，判断是否存在"登录态"特征（如聊天输入框、用户头像等登录后才有的元素；或反之存在登录墙/登录按钮）。无需额外弹窗或单独的登录界面。
- **未登录处理**：**不做任何特殊跳转**——webview 本就显示着该 AI 网站，未登录时它显示的就是该网站自己的登录页，**让用户直接在这个 webview 里主动登录**即可，登录态（cookie）随之保存。
- **状态反馈**：控制条胶囊显示「已登录（绿）/ 未登录（红）」；webview 顶部地址栏旁显示「已登录/未登录」小标。
- **门控**：两个选中的 AI 均检测为"已登录"时，`开始辩论` 才可点击。
- **会话失效**：辩论过程中若检测到某 AI 掉登录，暂停辩论并提示，用户在对应 webview 重新登录后可继续。

## 4.6 AI 适配器（逐站 DOM 差异的收口）

每个 AI 网页的注入方式、回答结构、完成信号、登录特征都**不同，需一站一分析**。把这些差异收进一个统一接口的"适配器"，每接一个 AI = 实现一个适配器；它是本项目的**主要维护面**（站点改版只改对应适配器）。可在 ChatALL 各 bot 接入代码基础上扩展。

适配器钩子：

| 钩子 | 作用 | 说明 |
|---|---|---|
| `isLoggedIn()` | 读页面 HTML/DOM 判断登录态 | 见 §4.5 |
| `newChat()` | 开新对话，清空旧上下文 | 每场辩论开始前调用，避免上下文污染 |
| `inject(text)` | 填入输入框并发送 | 注意 contenteditable / React 受控输入 / 回车 vs 发送按钮 |
| `latestAnswer()` | 读取最新一条回答文本 | 去除 UI 噪声（复制、重写等按钮） |
| **`isComplete()`** | **回答完成判定（核心）** | 基于 DOM 信号：如"停止生成"按钮出现→消失、发送按钮 disabled→enabled、回答容器在 N ms 内不再变化等；**逐 AI 不同**（Kimi、豆包等各有可观测信号） |
| `abnormalState()` | 检测异常并分类：**瞬时**(高峰/容量不足/限流/网络抖动) vs **硬错误**(登出/验证码/拒答) | **必须靠 DOM 结构判断，严禁文字匹配错误文案**（文案随时会变）。瞬时 → 自动重试(带退避)；硬错误 → 暂停并提示。绝不把错误内容当发言传递 |

**完成检测是整个辩论循环的命门**：未答完就读取会把半句话传给对方。`isComplete()` 必须可靠，且优先以"流式停止"为准。

**实测教训（SPIKE）：**
1. **完成检测竞态**：注入后须先确认"生成真正开始"（如停止按钮出现）再判结束，否则会在生成前因"短暂无变化"误判完成、读到残值。
2. **高峰容量错误**：Kimi 高峰会回"Kimi 有点累了…稍后再问"（限流），它会混进对话流。必须**按 DOM 结构识别**（如错误节点/重试控件），**不可匹配该文案**；识别后按"瞬时错误"重试。
3. 对"可疑过短的回答"应抓取该助手节点 DOM 以发现错误结构（见 `spike/`）。

## 4.7 辩手与会话（含同厂商）

- **立场靠 prompt 区分，而非靠不同模型**：正反双方可以是同一个 AI（默认两侧均为 DeepSeek）。
- **每侧各开独立对话线程**：两个 webview 各自 `newChat()`，互不串台；立场由注入的开场 prompt 决定。**同厂商时尤其要保证两条对话彼此隔离**：
  - `newChat()` 不仅"新建对话"，还要**锁定该对话的唯一标识**（如导航到 `…/chat/s/<id>` 并钉死），之后两侧各自只在自己的对话里收发。
  - **风险**：同一账号在两个 webview 中，站点可能把"当前对话"在标签页间**强制同步**（后建覆盖先建 → 正反方上下文串台）。须逐站在适配器中验证。
  - **兜底**：若该站强制同步，则同厂商两侧改用**独立 session 分区**（代价：需各自登录一次），或退化为**串行问答**（一侧答完再切另一侧）避免并发串台。
- **同厂商的会话隔离**：默认两个 webview 用同一 session/partition **共享登录（登录一次两侧通用）**，并配合上面的对话锁定保持独立对话。
- **记住上次选择**：持久化用户上次选的正方 / 反方 / 回合数，下次启动恢复；首次默认 DeepSeek × DeepSeek、5 回合。
- **注意事项**：
  - 同账号双线程 → 该账号**请求量翻倍，更易触发限流**（命中由 §4.6 `abnormalState()` 处理）。
  - 需验证目标站点**支持并发多对话**、且一个 webview 的"新建对话"不影响另一个（逐站在适配器中确认）。

## 4.8 辩论运行时交互（运行态）

1. **流式实时镜像（边吐边显）**：AI 在网页中流式输出时，左侧「辩论记录」**同步逐字更新**，而非等整段完成才显示。适配器的 `latestAnswer()` 需支持**反复读取当前（可能未完成）文本**，配合 `isComplete()` 判定收尾。
2. **进行中锁定**：辩论进行时**锁定页面切换/退出**（禁止切回页面一、禁止 webview 自行跳转/刷新）；要中断须经"停止"按钮（或退出前确认），避免把正在跑的对话搞断。
3. **单轮超时 + 异常检测**：
   - 每轮设超时（如 60s 未完成）→ **自动重试一次**；仍失败 → 暂停并提示。
   - 运行中持续检测**异常态**（§4.6，按 DOM 结构、非文字）：
     - **瞬时**（高峰/容量不足/限流/网络）→ 自动重试（带退避），不把错误内容当发言。
     - **硬错误**（会话失效/被登出/验证码）→ 暂停辩论 + 提示用户处理（如重新登录）。

4. **开辩前确认**：点「开始辩论」后先弹确认（如"正方=DeepSeek(支持) / 反方=DeepSeek(反对) / 5 回合，确认开始？"），用户确认后再进入页面二开跑——因为辩论要等数分钟，先确认设置无误。
5. **复制全文**：页面二提供「复制全文」按钮，一键复制整场辩论记录文本，便于分享。

> 说明：「每轮重申立场（防止越辩越附和）」**第一期不做**，留待后续评估。

原型见 `mockup/debate-ui.html`（浏览器打开，顶部步骤可切换）。

> **关键认知**：完全不调 API、纯白嫖网页版，所以辩论内容**不是我们自绘的气泡**，而来自**真实嵌入的 AI 网页**（webview）。我们的程序只往里"注入文字"、"读取回答"，再汇总展示。

分两个页面（步骤）：

**页面一 · 准备 / 登录**
- 控制条：话题输入框 +「开始辩论」（门控）；胶囊选择「正方 AI / 反方 AI / 回合数」，胶囊上显示各 AI 登录状态。
- 登录区：两个 webview 加载选中的 AI 真实网页。已登录显示就绪；未登录则该 webview 显示该网站自己的登录页，用户直接在此登录（见 §4.5）。
- 两个 AI 均检测为已登录后，「开始辩论」点亮 → 进入页面二。

**页面二 · 辩论**
- 顶部：辩题 + 回合进度（如"第 2 / 5 回合"）+ 停止/重设。
- 「辩论记录」（主视图）：把两个网页抓到的发言按回合拼成一场干净的正反交锋（正方 / 反方分色）；结尾"— 辩论结束 · 共 N 回合 —"。这是收集双方消息的聚合视图——两个 AI 在各自网页里互相看不见，必须由此汇总。
- 「网页版实况」（后台引擎）：两个 webview 缩小展示，可见程序注入与 AI 实时回答，退居其次。

- 风格：浅色玻璃拟态、柔和阴影、Keynote 质感。

## 7. 风险与代价（白嫖固有）
1. **维护成本**：AI 网页改版会使注入选择器失效→自动化中断，需跟随修复（fork 活跃开源项目可借社区之力）。
2. **登录要求**：豆包 / DeepSeek / ChatGPT 等需用户在 App 内登录自己账号。
3. **ToS / 反爬**：自动化操作网页版属灰色地带，站方可能上验证码打断。
4. **绕圈**：固定回合 + 明确正反方立场 + 每轮注入对方原话，可大幅缓解两个 AI 互相空附和。

## 8. 第一期范围（MVP）边界
**做：** 选两个 AI（含同厂商，默认 DeepSeek×DeepSeek，记住上次选择）、输入话题、自动分正反方、固定 N 回合（默认 5）自动辩论、Keynote 风格两页流程、提示词按语言环境自动生成、登录检测与门控（开辩前 + 辩论中）、AI 适配器（注入/读取/完成检测/异常检测）、辩论运行时交互（边吐边显、进行中锁定、单轮超时重试 + 异常检测）。

**做（补充）：** 开辩前确认、复制全文。

**不做（后续阶段再议）：** 裁判/总结、每轮重申立场、用户自定义提示词与人设、导出/分享卡片（含图片/链接）、超过两个 AI 的多方辩论、手机端。
